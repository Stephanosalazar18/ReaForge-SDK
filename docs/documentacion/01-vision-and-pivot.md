# ReaForge — Vision and June 2026 Pivot

## What ReaForge is

A REAPER extension whose purpose is **letting a user generate REAPER artifacts via an AI agent in natural language**.

The agent only **generates and persists files**. It does not mutate REAPER's live state. The user runs the generated artifacts when they want.

### Concrete examples (from the user)

| Request | Artifact | Saved at |
|---|---|---|
| "Create a spectral morph effect between these two sounds" | `.jsfx` | `REAPER/Effects/ReaForge/spectral_morph.jsfx` |
| "Combine Cockos delays into a new delay effect" | `.RfxChain` | `REAPER/FXChains/ReaForge/custom_delay.RfxChain` |
| "Write a script that humanizes MIDI velocity on the selected items" | `.lua` + registered action | `REAPER/Scripts/ReaForge/humanize.lua` |
| "Open the automation envelope for parameter X on track 3" | — | **Out of MVP** (agent does not mutate live state) |

Everything the agent saves persists in REAPER's filesystem and works without the extension running.

### Target user

A REAPER user (producer, sound designer, composer). Not a developer. They want to **describe an outcome** in natural language and get a usable REAPER artifact.

## What ReaForge is NOT

These were explicit rejections from the 2026-06-07 conversation:

| Wrong scope (originally built) | Rejected because |
|---|---|
| A multi-runtime SDK that embeds Lua + JSFX + QuickJS in one C++ DLL | REAPER already runs `.lua` and `.jsfx` natively. The extension only needs to write files, not execute them. |
| A platform for third-party developers to ship extensions | The target user is a producer, not a dev. "Extensions" are generated by the AI on demand, not authored by humans. |
| An agent that mutates live REAPER state (changes parameters, opens automation, triggers playback) | Out of MVP. The agent only generates artifacts. The user runs them. |
| Three flows shipped in order (native panel → opencode CLI → embedded opencode dock) | Native panel and embedded dock deferred to the **last** phase. MVP is opencode desktop only. |

## The pivot (2026-06-07)

The user corrected two scope errors in a single message:

> "el agente solo genera codigo, tarea, scripts y los guarda en reaper"
> *(the agent only generates code, tasks, scripts and saves them in REAPER)*

> "todo esto se guardaria directamente en reaper para poder usarse cuando sea, osea miniapps generadas con segun el lenguaje y guardadas donde le corresponderia"
> *(all of this would be saved directly in REAPER for use whenever, i.e. mini-apps generated according to the language and saved where they belong)*

### What changed

| Was | Is now |
|---|---|
| Multi-runtime SDK (Lua + JSFX + QuickJS embedded in DLL) | Agentic artifact generator (extension only writes native REAPER files) |
| Agent mutates live REAPER state | Agent only reads context + writes files + asks REAPER to refresh its caches |
| 3 flows shipped sequentially | 1 flow (opencode desktop) for MVP. Native panel + context menu deferred to last phase. |
| Bridge tools mostly imperative (`run_extension`, `list_tracks`) | Bridge tools split: Read (context) / Write (artifacts) / Refresh |

## Decisions that result from the pivot

| Decision | Rationale |
|---|---|
| Agent runs **outside** REAPER, in `opencode desktop` | Reuses opencode's UI, conversation history, multi-model support, tool approval, MCP transport. Avoids 3–6 months of building an LLM client + chat UI in C++. |
| Bridge tools change: from "execute" to "read context + write artifacts + refresh REAPER" | Matches the new vision. The 5 original tools (`list_tracks`, `run_extension`, …) get retired or repurposed. New shape: 3 Read + 3 Write + 1 Refresh = 7 tools. |
| C++ extension scope shrinks: HTTP server + filesystem writes + refresh hooks | No runtime embedding. Smaller surface, less risk, faster to ship. |
| Internal ReaImGui panel + context menu hooks deferred to the **last** phase | MVP only needs opencode desktop → bridge → extension. UI inside REAPER is a UX improvement, not a validation requirement. |
| Existing pre-pivot docs (`docs/cross-environment.md`, `docs/user-flows.md`) stay as-is | They describe the pre-pivot architecture. The transport layer (HTTP + `wsl-bridge.txt`) is **still valid**. The Flow 1/2/3 model is superseded by this folder. |

## How to validate the new vision

A single-sentence success criterion:

> The user opens opencode desktop, types one of the example prompts above, and within 2 minutes a working JSFX/Lua/RfxChain file appears in the right REAPER folder, ready to use.

The MVP scope and the next concrete step are in [`04-mvp-and-handoff.md`](./04-mvp-and-handoff.md).
